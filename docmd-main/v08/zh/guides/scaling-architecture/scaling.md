## 问题

随着软件的成熟，文档自然会扩展。当项目发展到数千个 Markdown 文件时，传统的生成器会受到影响。构建时间拖慢，热重载变慢，导航结构让维护者和用户都感到不知所措。

## 为什么重要

如果构建需要数分钟，作者会避免进行小的更正。内容会变得陈旧和不准确。对于用户来说，一个庞大的、组织混乱的侧边栏菜单会使查找信息变得令人沮丧。这会增加支持工单并降低开发体验。

## 方法

docmd 为速度和可扩展性而构建。通过使用高性能解析引擎和细粒度的基于文件的策略，它能在几秒钟内处理数千个页面。其优化的 SPA 交付确保了最终用户的导航保持即时性。

## 实现

### 1. 细粒度项目结构

避免扁平的目录。使用深度嵌套的文件夹结构来镜像您产品的架构。这简化了维护，并允许 docmd 在开发过程中高效地跟踪更改。

### 2. 优化的搜索索引

对于大型站点，[搜索插件](../../plugins/search.md) 是必需的。docmd 生成高度压缩的搜索索引，按需加载。这在提供跨数千页面的全文搜索的同时保持了快速的初始页面加载。

### 3. 版本控制与归档

使用 [版本管理引擎](../../configuration/versioning.md) 来分隔遗留内容。将旧版本隔离到不同的构建上下文中，可减少日常更新期间处理的页面数量。这提高了开发速度。

```json "docmd.config.json"
  "versions": {
    "current": "v3",
    "all": [
      { "id": "v3", "dir": "docs/current", "label": "v3.x (Latest)" },
      { "id": "v2", "dir": "docs/v2",      "label": "v2.x (Legacy)" }
    ]
  }
```

### 4. 基于组件的导航

使用 `navigation.json` 文件将导航分解为逻辑段。这让您可以为不同部分定义不同的侧边栏层级，防止主导航变得杂乱。

## 权衡

大型站点自然会消耗更多磁盘空间和构建期间的内存。要在极端规模（10,000+ 页面）下保持亚秒级构建时间，请使用配备 SSD 存储和充足 RAM 的高性能 CI/CD 环境来处理并行文件处理。
