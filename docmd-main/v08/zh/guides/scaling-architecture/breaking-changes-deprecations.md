title: "破坏性变更与弃用"
description: "如何使用版本化文档和上下文标注有效地传达 API 变更和迁移路径。"
---

## 问题

当产品引入重大版本变更时，API 或配置不可避免地被弃用。浏览最新文档的用户如果使用过时的模式必须得到警告。但是，文档应保持专注于现代实现。

## 为什么重要

未能突出显示破坏性变更会导致开发人员调试引擎不再支持的代码。上下文警告和清晰的迁移路径可维护用户信任、减少支持请求，并确保平稳过渡。

## 方法

将 docmd 的 [版本管理引擎](../../configuration/versioning.md) 与 [标注容器](../../content/containers/callouts.md) 相结合。这在遗留内容和现代内容之间创建了明确的区分。在当前版本中提供"迁移锚点"，同时将完整遗留文档移至归档版本。

## 实现

### 1. 归档遗留内容

发布新的主要版本时，将现有文档移至归档目录（例如 `docs-v1/`）。这为尚未迁移的用户保留了上一版本的上下文。

### 2. 上下文迁移标注

在您的最新文档中，在发生重大变更的位置使用 `warning` 或 `important` 标注。这为尝试使用遗留模式的用户提供即时价值。

```markdown
# Configuration API

::: callout warning "迁移：v2.0 中的破坏性变更"
`siteTitle` 属性已移除。已由全局 `title` 属性取代。

* **从 v1.x 迁移？** 请更新您的 `docmd.config.json`。
* **需要最新文档？** 请参阅[配置指南](../../configuration/overview.md)。
:::
```

### 3. 维护 AI 准确性

将已弃用内容与活动版本分开，可提高 AI 工具的准确性。docmd 的 [LLMs 插件](../../plugins/llms.md) 根据活动版本生成上下文文件。归档遗留内容可防止 AI 模型推荐过时的 API。

## 权衡

主动管理迁移标注会增加维护开销。如果无限期保留，页面会因旧警告而变得杂乱。一旦遗留版本达到生命周期终止（EOL）或经过一个完整的主要发布周期后，移除迁移标注。这可使文档保持精简和专注。
