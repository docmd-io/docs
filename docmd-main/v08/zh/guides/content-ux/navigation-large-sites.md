---
title: "大型站点的导航"
description: "如何利用 docmd 的布局工具，把复杂的文档集合组织成直观、可扩展的导航结构。"
---

## 问题

当站点增长到几百个页面时，简单的侧边栏往往会变成层层嵌套的迷宫。用户要展开多层才能找到参考，会逐渐丢失上下文。感到受挫的用户会放弃文档，转而靠"试错"摸索。

## 为什么重要

导航是产品能力的"地图"。如果导航困难，用户只能完全依赖搜索栏，导致知识是碎片化的。良好的导航结构能让用户理解产品的分类体系，并随着时间变得更为独立。

## 方法

比起深度嵌套，应优先考虑 **顶层上下文切换 (Top-Level Context Switching)**。把左侧侧边栏的深度控制在两到三层之内。使用横向的 [菜单栏](../../configuration/menubar.md) 来区分不同的文档"域"（例如：指南、API 参考、社区）。这样每个侧边栏都能保持专注与可控。

## 实现

### 1. 按域拆分

在 `docmd.config.json` 中，通过 [菜单栏](../../configuration/menubar.md) 把内容划归高层级类别。这种做法让您能为每个域呈现完全不同的侧边栏，避免单一导航树失控。

### 2. 压平层级

与其把一个概念分散到许多细碎页面，不如把相关信息合并到内容更完整的父页面里。使用清晰的 [标题层级](../../content/syntax/index.md)，让用户能借助页面自动生成的右侧目录 (TOC) 在页内导航。

*   **❌ 不好的 IA**：一个名为 "Security" 的目录，里面塞了十份不同协议的一段落短文。
*   **✅ 更好的 IA**：一个结构良好的 "Security Overview" 页面覆盖所有协议，并通过标题生成清晰的目录。

### 3. 使用可折叠区

对于很少被访问、又彼此相关的内容组，可在 [导航配置](../../configuration/navigation.md) 中使用 `collapsible` 属性。它能让次要信息默认收起，只在用户主动展开时才出现。

```json "navigation.json"
// navigation.json
{
  "title": "API 参考",
  "collapsible": true,
  "collapsed": true,
  "children": [
    { "title": "鉴权", "path": "api/auth" },
    { "title": "端点", "path": "api/endpoints" }
  ]
}
```

## 取舍

把内容合并到更少、更长的页面里，要求作者在结构清晰度与标题使用上保持纪律。如果一个页面过长、又缺少良好的内部导航，就会变成另一种"文字墙"。然而，"点击疲劳"的显著减少以及发现度的提升，让扁平化、按域划分的层级更适合大型文档集合。
