### ✨ Highlights

Dieses Release liefert die **Custom Template Plugin API** — eine vollständig modulare Möglichkeit, alternative Layouts als eigenständige Plugins auszuliefern — vorgestellt von **Summer**, dem ersten offiziellen Community-Template (installierbar über den neuen Befehl `docmd add summer`). Zwei optionale **Core-UI-Features** (Cookie-Zustimmungsdialog und seitenweite Ankündigungsleiste) sind nun in `@docmd/ui` eingebaut, sodass jedes Template sie ohne separates Plugin übernehmen kann. Außerdem enthält dieses Release ein neues Asset-Prioritätssystem, einen **Fix für den Docker-`:latest`-Tag** (Quickstart funktioniert nun out of the box) und keinerlei Breaking Changes.

## 🧩 Custom Template Plugin API

Templates sind nun Plugins. Ein Template-Paket deklariert `capabilities: ['template']` und liefert ein `templates[]`-Array mit `.ejs`-Datei-Overrides. Der Resolver in `@docmd/ui` (neuer `resolveTemplate()`) beachtet diese Auflösungskette:

1. `frontmatter.template` (pro Seite)
2. `config.templates[glob]` (pro Sektion, zukünftig)
3. `config.theme.template` (seitenweit)
4. Eingebauter Standard aus `@docmd/ui`

Schlägt ein Schritt fehl oder fehlt die aufgelöste Datei auf der Festplatte, fällt der Resolver auf den Standard zurück und gibt eine einzelne TUI-Warnung aus. Templates liefern ihre eigene CSS/JS über `templateAssets[]`, die mit Priorität `10` geladen werden, sodass benutzerdefiniertes `customCss` (Priorität `15`) immer gewinnt. Templates **dürfen** in CSS **kein** `!important` verwenden.

**Auflösungsbeispiel**

```js
// template-summer/index.js
export default {
  plugin: {
    name: 'template-summer',
    version: '0.1.0',
    capabilities: ['template'],
  },
  templates: [
    { type: 'layout', templatePath: '...' },
    { type: 'sidebar', templatePath: '...' },
  ],
  templateAssets: [
    { type: 'css', path: '.../summer.css', priority: 10, position: 'head' },
  ],
};
```

```json "docmd.config.json"
// docmd.config.json
{ "theme": { "template": "summer" } }
```

Templates können auch jeden der 14 unterstützten Slots weglassen (`layout`, `404`, `toc`, `navigation`, `sidebar`, `header`, `footer`, `menubar`, `options-menu`, `project-switcher`, `version-dropdown`, `language-switcher`, `banner`, `cookie-consent`); der Resolver fällt für nicht bereitgestellte Slots automatisch auf den Standard zurück.

**Pro-Seite-Override über Frontmatter**

```markdown
---
title: "Changelog"
template: "template-changelog"
---
```

Das erste offizielle Template-Paket — `@docmd/template-summer` (das **Summer**-Layout) — wird synchron mit diesem Release ausgeliefert und ist über die Standard-`docmd add`-Pipeline installierbar.

## ☀️ Summer-Template-Vorschau

Das erste offizielle Template auf Basis der Custom Template API ist **Summer** — ein helles, hoffnungsfrohes, sommerliches Layout mit oberer Suchleiste, weichen Halo-Akzenten und dichterem Karten-Raster. Installation über die Standard-`docmd add`-Pipeline:

```bash
docmd add summer
```

Dies installiert `@docmd/template-summer` aus npm und aktualisiert automatisch Ihre `docmd.config.json`:

```diff
  module.exports = defineConfig({
+   theme: {
+     template: "summer"
+   },
    // ... Ihre bestehende Konfiguration
  });
```

Im Gegensatz zu normalen Plugins schreibt Summer in `theme.template` (nicht `plugins.*`) und **ersetzt** ein zuvor gesetztes Template — es gibt kein Stapeln. Führen Sie `docmd remove summer` aus, um zum Standard zurückzukehren. Summer liefert eine eigene `404.ejs`; wenn dieser Slot weggelassen wird, dient die Standard-404 als Fallback.

> **Release-Reihenfolge:** `@docmd/template-summer@0.8.6` wird vor dem Rest des Monorepos veröffentlicht, um die OIDC-First-Publish-Anforderung von npm zu erfüllen (die vom lokalen Rechner des Maintainers ausgehen muss). Die synchron mit `0.8.7` veröffentlichte Version ist funktional identisch.

## 🍪 Cookie-Zustimmungsdialog

Ein minimalistischer, barrierefreier Cookie-Zustimmungsdialog ist nun Teil des Standard-UI. Aktivieren Sie ihn in Ihrer Konfiguration — keine Plugin-Installation erforderlich.

```json "docmd.config.json"
{
  "cookie": {
    "enabled": true,
    "message": "We use cookies to ensure you get the best experience.",
    "policyUrl": "/privacy",
    "position": "bottom-right",
    "dismissible": true,
    "expiryDays": 180
  }
}
```

| Feld | Standard | Beschreibung |
|---|---|---|
| `enabled` | `true` (sofern `cookie`-Objekt vorhanden) | Hauptschalter. |
| `message` | Übersetzungsschlüssel `cookieMessage` | Text des Dialogs. |
| `acceptText` | Übersetzungsschlüssel `cookieAccept` | Beschriftung der Akzeptieren-Schaltfläche. |
| `declineText` | Übersetzungsschlüssel `cookieDecline` | Beschriftung der Ablehnen-Schaltfläche. |
| `policyUrl` | `null` | Optionaler Link zu Ihrer Datenschutzerklärung. |
| `position` | `bottom` | `bottom` \| `bottom-left` \| `bottom-right` \| `center` |
| `dismissible` | `true` | Schließen-Schaltfläche (X) anzeigen. |
| `expiryDays` | `180` | Wie lange die Auswahl in `localStorage` gespeichert bleibt. |

Wenn `config.cookie` nicht gesetzt ist, wird **kein Dialog gerendert** — Opt-in by Design. Nach einer Auswahl wird auf dem Window ein `docmd:cookie-consent`-`CustomEvent` ausgelöst, auf den Templates und Plugins reagieren können.

## 📢 Seitenweite Ankündigungsleiste

Eine schließbare Leiste, die über der Menubar und unter dem Seiten-Header sitzt. Konfigurieren Sie Inline-Markdown (`**fett**` und `` `code` ``) oder übergeben Sie rohes `html`.

```json "docmd.config.json"
{
  "layout": {
    "banner": {
      "content": "**v0.9 erscheint am Freitag** — lesen Sie die Ankündigung.",
      "type": "info",
      "dismissible": true,
      "link": { "text": "Mehr lesen", "url": "/blog/v0-9" },
      "icon": "megaphone"
    }
  }
}
```

| Feld | Standard | Beschreibung |
|---|---|---|
| `content` | `""` | Inline-Markdown-Text. Schließt sich gegenseitig mit `html` aus. |
| `html` | `""` | Roher HTML-Code. Hat Vorrang vor `content`. |
| `type` | `"info"` | `info` \| `success` \| `warning` \| `danger` |
| `dismissible` | `true` | Schließen-Schaltfläche (X) anzeigen. |
| `link` | `null` | `{ text, url }` für einen CTA-Link. |
| `icon` | `null` | Lucide-Iconname, der links angezeigt wird. |

Der "geschlossen"-Zustand wird in `sessionStorage` gespeichert, sodass die Leiste bei einem neuen Besuch wieder erscheint. Wenn `config.layout.banner` nicht gesetzt ist, wird **keine Leiste gerendert** — Opt-in by Design.

## Changelog

### Neue Funktionen

- **Custom Template Plugin API** — Templates sind nun Plugins mit `capabilities: ['template']`, `templates[]` und `templateAssets[]` Hooks; neuer `resolveTemplate()`-Resolver mit 4-stufiger Fallback-Kette (pro Seite → pro Sektion → seitenweit → Standard).
- **Neue Typen** in `@docmd/api` und `@docmd/ui`: `Capability: 'template'`, `TemplateHook`, `TemplateAssetHook`, `TemplateSlot`, `TemplateResolutionContext`, `ResolvedTemplate`, `Asset`, `AssetKind`, `AssetPosition`.
- **Neue Registry-Felder** auf `PluginHooks`: `templates` und `templateAssets`; neue `PluginModule.templates?` / `templateAssets?` Exports.
- **Neue Resolver-API** in `@docmd/ui`: `resolveTemplate()` und `clearTemplateResolverCache()`.
- **`docmd add summer`** — erstklassige Template-Installation, die in `theme.template` schreibt (nicht `plugins.*`); `docmd remove summer` entfernt sie.
- **Cookie-Zustimmungsdialog** in `@docmd/ui` eingebaut — Opt-in via `config.cookie`; emittiert `docmd:cookie-consent`-`CustomEvent`.
- **Seitenweite Ankündigungsleiste** in `@docmd/ui` eingebaut — Opt-in via `config.layout.banner`; Schließzustand in `sessionStorage` gespeichert.
- **Asset-Prioritätssystem** — deterministische CSS/JS-Ladereihenfolge: Basis(0) → Theme(5) → Template(10) → customCss(15) → Plugins(20).
- **14 unterstützte Template-Slots**: `layout`, `404`, `toc`, `navigation`, `sidebar`, `header`, `footer`, `menubar`, `options-menu`, `project-switcher`, `version-dropdown`, `language-switcher`, `banner`, `cookie-consent`.

### Fehlerbehebungen

- **Docker `:latest`-Tag fehlt** — das veröffentlichte Image hatte keinen `latest`-Tag, da der `enable`-Ausdruck des Workflows ein `push`-Event auf `main` verlangte, der Workflow aber keinen `push`-Trigger hatte. Alle Dokumentation auf `:0.8.6` festgepinnt und [`.github/workflows/docker-publish.yml`](https://github.com/docmd-io/docmd/blob/dev-0.8.7/.github/workflows/docker-publish.yml) umgearbeitet, um (a) bei `release`-Events automatisch `:latest` zu taggen, (b) einen `push: branches: [main]`-Trigger für `:edge`-Rebuilds erneut hinzuzufügen, (c) ein manuelles `tag_latest`-Input zum Nachtaggen bereitzustellen. Vollständiger Kontext im [Bug-Report-Audit](https://github.com/docmd-io/docmd/blob/dev-0.8.7/bug-report.md).
- **i18n-Strings in Partials nicht übersetzt** — `menubar`, `options-menu`, `version-dropdown`, `language-switcher`, `project-switcher` und `toc` Partials verwendeten `t('…')`-Aufrufe, bekamen aber die `t`-Funktion nie vom Layout übergeben (nur `banner`, `footer` und `cookie-consent`). `t: typeof t === 'function' ? t : null` zu jedem `include()`-Aufruf in `layout.ejs` hinzugefügt, sodass nun alle Partials korrekt übersetzen. Außerdem 13 fehlende Übersetzungsschlüssel (`close`, `cookieAccept`, `cookieConsent`, `cookieDecline`, `cookieMessage`, `cookiePolicy`, `goBack`, `lastUpdated`, `pageNotFoundMsg`, `readMore`, `recentCommits`, `searchPlaceholder`, `selectProject`) zu allen 7 Locale-Dateien (`en`/`de`/`es`/`fr`/`hi`/`ja`/`zh`) hinzugefügt.
- **Code-Block-Kopierschaltfläche schwebte über dem Block** — die Schaltfläche war an einem inneren `.code-wrapper` verankert (über Inline-`style.position = 'relative'` in `docmd-main.js`), das sie nur oben rechts am `<pre>` platzierte und die Titelleiste darüber ignorierte. Verankert nun am äußeren `.docmd-code-block-wrapper` (das in CSS nun `position: relative` ist), sodass sie sauber relativ zum gesamten Code-Block inklusive Titel positioniert ist. Außerdem wird die redundante innere Wrapper-Erstellung für Code-Blöcke übersprungen, die bereits einen vom Parser gerenderten Wrapper haben.
- **Summer-Template: großer Abstand unter kurzem Inhalt** — `.summer-content` hatte `flex: 1 0 auto`, wodurch der Artikel die Rasterzelle füllte und auf kurzen Seiten eine riesige Lücke zwischen Artikel und Footer ließ. Das Grow entfernt; der Inhalt hat nun seine natürliche Höhe, der Footer haftet weiterhin über das bestehende `margin-top: auto` am unteren Rand.
- **Summer-Template: Kopierschaltfläche fehlt in Code-Block-Titelleiste** — `summerCodeblocks()` in `summer.js` suchte `.copy-code-button` nur innerhalb von `.code-wrapper`, aber `injectCopyButtons()` in `docmd-main.js` hängt die Schaltfläche für Code-Blöcke mit Parser-gerendertem Titel (z. B. ```` ```js "main.ts" ````) direkt an `.docmd-code-block-wrapper` an (ohne inneren `.code-wrapper`). Die Schaltfläche blieb inline unter dem `<pre>` statt in die Titelleiste zu wandern. `summerCodeblocks()` sucht nun im Wrapper selbst nach der Schaltfläche, sodass die Titelleiste in beiden Code-Block-Pfaden das Kopier-Control erhält.
- **Summer-Template: Changelog-Zeitlinie an Eintragspunkten versetzt** — zwei CSS-Regeln für `.changelog-timeline` (eine spezifische, eine generelle) überschnitten sich teilweise: die spezifische Regel setzte die Linie auf `left: 0`, die generelle Regel fügte dem Zeitlinien-Container aber `padding-left: 32px` hinzu und schob jeden Eintrag 32px nach rechts. Der Punkt schwebte 28px neben der Linie. Der Linien-Offset der spezifischen Regel ist nun `left: 32px`, sodass das Linienzentrum mit dem Punktzentrum jedes Eintrags fluchtet.
- **Docker-Image: Quickstart brach mit "No documentation directory or configuration found" ab** — `WORKDIR` des Images war `/docs` und der Standard-`CMD` war `docmd dev --host 0.0.0.0`, aber `/docs` war im Image leer (das Template-Saatgut lag in `/template`). Ein kleines `docker-entrypoint.sh` hinzugefügt, das `/template` nur dann nach `/docs` kopiert, wenn `/docs` vollständig leer ist (sodass benutzerseitig gemountete Volumes nie überschrieben werden), und anschließend den übergebenen Befehl `exec`-utet. Das Image funktioniert nun out of the box ohne Volume-Mount.
- **Docker-Image: Healthcheck meldete stets "unhealthy"** — das `HEALTHCHECK` des Images war hartcodiert `curl http://localhost:3000/`, aber `docmd dev` inkrementiert automatisch auf 3001+, wenn 3000 belegt ist (z. B. wenn bereits ein anderer Dev-Server auf dem Host läuft). Inline-curl durch `docker/docker-healthcheck.sh` ersetzt, das 3000–3005 prüft und 0 zurückgibt, sobald ein beliebiger Port antwortet.
- **Bedingtes Plugin-Asset-Laden brach SPA-Navigation** — wenn eine Seite das `<script src="...">`-Tag eines Plugins bedingt ausgab (z. B. Mermaids `init-mermaid.js` nur auf Seiten mit `class="mermaid"`), wurde das Skript-Tag **nach** `</main>` in `<body>` gerendert, sodass der Content-Swap der SPA-Navigation (der nur `.main-content` und eine feste Selektorliste ersetzt) das neue Tag im geparsten `doc` zurückließ, es aber nie zum Live-Dokument hinzugefügt wurde. Navigation von einer Nicht-Mermaid-Seite zu einer Mermaid-Seite ließ die `.mermaid`-Divs ungerendert (kein Listener, kein IIFE, kein Init). Direktes Laden und Reload funktionierten, weil das Skript bereits im initialen HTML war; KaTeX (nur CSS, in `<head>`) funktionierte, weil die bestehende Head-Asset-Synchronisierung es handhabte. Body-Skript-Sync in `navigateTo()` hinzugefügt, der den Head-Sync spiegelt: für jedes `body > script[src]` im neuen HTML, das noch nicht im Live-Dokument vorhanden ist, wird ein frisches `<script>`-Element (kein `cloneNode` — Browser überspringen geklonte Skripte) mit der aufgelösten absoluten URL erzeugt und angehängt. Das IIFE des Plugins handhabt das Rendering der aktuellen Seite (ruft sein Setup direkt auf, wenn `document.readyState !== "loading"`); der `docmd:page-mounted`-Listener, den es registriert, deckt nachfolgende Navigationen ab. Die Synchronisierung ist vollständig generisch — keine Plugin-Namen, keine Schlüsselwörter, keine hartcodierten Pfade — und funktioniert daher mit jedem zukünftigen Plugin-Asset.
- **Summer-Template: betitelte Code-Blöcke verloren Padding und Schriftart** — der Selektor `.code-wrapper>pre, .docmd-code-block-wrapper>.code-wrapper>pre` passte nur auf Code-Blöcke mit einem inneren `.code-wrapper` zwischen dem äußeren Wrapper und dem `<pre>`. Betitelte Code-Blöcke (`` ```js "title" `` `) haben `<pre>` als direktes Kind von `.docmd-code-block-wrapper` — ohne inneres `.code-wrapper`, da `injectCopyButtons` die Kopierschaltfläche direkt an den Wrapper anhängt. Der Selektor verfehlte diese `<pre>`s, sodass sie null Padding und die monospace Standardschrift des Browsers erbten, statt Summer-Tokens (Code wurde gegen die Wrapper-Kanten gequetscht). `.docmd-code-block-wrapper>pre` zu den pre- und pre-code-Selektoren hinzugefügt, sodass alle drei Code-Block-Formen (`.code-wrapper > pre`, `.docmd-code-block-wrapper > .code-wrapper > pre`, `.docmd-code-block-wrapper > pre`) Summers Padding (16px 18px), JetBrains-Mono-Schrift (0.84rem), Zeilenhöhe (1.65) und Vordergrundfarbe erhalten.
- **Summer-Template: Code-Block-Regeln enthielten übriggebliebenes `!important` und verletzten die Template-Richtlinie** — der Datei-Header selbst erklärt "Do NOT use `!important`. User customCss (priority 15) muss immer gewinnen, damit Nutzer jeden Selektor ohne Fork des Templates neu stylen können", aber die pre/code-wrapper-Regeln hatten `!important` auf 8 Eigenschaften. Das `!important` war ein Überbleibsel aus der Zeit, als `docmd-main.css` und `summer.css` gemeinsam geladen wurden; im aktuellen Design lädt das Summer-Layout `docmd-main.css` **nicht** (gemäß Header-Kommentar des Layouts), und Summers Selektoren haben alle eine höhere Spezifität als die Basis-`pre`-Regel, sodass `!important` nicht mehr nötig war. Aus den Code-Block-pre-, code-wrapper- und pre-code-Regeln entfernt. Empirisch verifiziert: Summers Stile gelten weiterhin (Padding, Schriftart, transparenter Hintergrund), und benutzerdefiniertes `customCss` kann Summer nun korrekt überschreiben, ohne `!important` zu benötigen (wie von der Richtlinie vorgesehen) — während `customCss` mit `!important` weiterhin gewinnt (wie es sein sollte).
- **Sidebar-Navigation: Auf-/Zuklappen bei echten Links still defekt** — `navigation.ejs` platziert `.nav-group` sowohl auf dem Dummy-`<span class="nav-label">` als auch auf dem echten `<a class="nav-group">` (mit `href`). Der globale Click-Handler passte nur auf `.nav-label, .collapse-icon-wrapper`, sodass ein Klick auf den Label-Text eines echten Links (z. B. "Konfiguration") still nichts tat — nur der Chevron funktionierte. Toggle-Handler aktualisiert, sodass er auch `.nav-group` matcht; ein Klick auf den Label-Text klappt die Gruppe nun auf/zu. Damit Toggle-Handler und SPA-Router nicht beide feuern (was einfach wegnavigieren und den Sidebar-Zustand auf der neuen Seite zurücksetzen würde), überspringt der SPA-Router nun `<a class="nav-group">` innerhalb von `li.collapsible`. Auf Playground (Summer-Template) und Doku-Seite (Standard-Theme) verifiziert: zugeklappt → Klick auf `<a>` → Gruppe klappt auf, keine SPA-Navigation; aufgeklappt → Klick auf `<a>` → Gruppe klappt zu (aria-expanded=false, Untermenü-Animation schließt); Blattlinks (ohne `.nav-group`) navigieren weiterhin per SPA wie zuvor.
- Asset-Pipeline verwendet nun eine stabile Prioritätssortierung und beseitigt die nicht-deterministische CSS-Ladereihenfolge, wenn mehrere Plugins gleichzeitig Stile ausgeben.
- `loadPlugins()` setzt die `templates`- und `templateAssets`-Arrays nun zwischen Builds zurück, entsprechend dem Lebenszyklus aller anderen Hook-Arrays.
- `template-resolver.ts` normalisiert Plugin-Namen mit `@docmd/`- und `template-`-Präfixen gleichermaßen (`summer`, `template-summer`, `@docmd/template-summer` lösen identisch auf).
- Neues `Asset`-Interface typprüft den zuvor `any[]`-Rückgabewert von `getAssets()`.

## 🛣️ Ausblick: 0.9.0

> **Hinweis für nachgelagerte Konsumenten:** Die nächste Minor-Version (0.9.0) erfordert **Node.js 20+** als minimale unterstützte Laufzeit. Die gesamte Plattform bewegt sich zusammen, sodass Engines, Types, ausgelieferte Laufzeit und Build-Umgebung konsistent sind.

Das 0.9.0-Ziel (Node 22 als globaler Floor):

- `engines.node: ">=20"` — minimale Node-Version, die Benutzer benötigen (Node 18 fällt)
- `@types/node: "^22"` — Dev-Time-TS-Ziel (passt zur CI)
- `Dockerfile` Basis-Image: `node:22-alpine` — ausgelieferte Laufzeit
- CI `node-version: '22'` — Build-Umgebung

Warum dies jetzt kommt:

- Node 18 ist seit April 2025 EOL, Node 20 seit April 2026. Beide erhalten keine Upstream-Sicherheitspatches mehr.
- Node 22 ist in Wartung (EOL April 2027) und gibt Benutzern rund 10 Monate Überlappung mit ihrem bestehenden Plattform-Standard. Es ist das faktische "vorherige LTS", das die meisten Enterprise-Flotte-Upgrades bereits eingeführt haben.
- Den Floor synchron mit der ausgelieferten Laufzeit anzuheben bedeutet, dass Image, Types und Engines-Versprechen alle übereinstimmen — das gleiche "Single-Number"-Prinzip, das wir für 0.8.7 übernommen haben, nur eine Hauptversion höher.

Die Roadmap nach 0.9.0:

- **1.0.0** — Node 22 fällt; Floor wird `>=22`, Types `^24` (passend zum neuesten LTS der CI), Docker `node:24-alpine`, CI `'24'`.

Dies ist ein einzelner 0.9.0-Commit, ~30 Datei-Änderungen (eine pro package.json plus Docker und Workflows), keine Quellcode-Änderungen. Die 0.8.7-Linie behält ihren aktuellen Floor (`>=18`) bis 0.9.0 ausgeliefert wird, sodass es für 0.8.x-Benutzer keine überraschenden Breaking Changes gibt.
