The `docmd` 0.7.7 release introduces **multi-project support** — the ability to build and serve multiple independent documentation sites from a single docmd instance. One command, one domain, independent projects.

This release also focuses on **developer experience** with a brand new internal "prep" system and graceful, stack-trace-free configuration validation.

## ✨ Highlights

### Multi-Project Configuration

A single `docmd.config.js` at the repository root can now define multiple documentation projects. Each project gets its own prefix, its own config, its own versioning, and its own navigation — but they all share the same theme, assets, and deployment pipeline.

```javascript
// docmd.config.js (root)
module.exports = defineConfig({
  projects: [
    { prefix: '/',       src: 'docmd-main' },
    { prefix: '/search', src: 'docmd-search' }
  ]
});
```

**Output:**
```
site/                     ← root project (docmd-main)
site/search/              ← prefixed project (docmd-search)
```

Each project folder contains its own `docmd.config.js` with full independent configuration — title, i18n, versions, plugins, navigation, layout, and everything else.

See the [Multi-Project Configuration](/configuration/multi-project/) guide for full details.

### Shared Assets

Assets placed at the repository root (`assets/`) flow automatically to all projects. Each project can also have its own `assets/` directory that takes priority over shared assets.

```
my-docs/
├── assets/                 ← shared across all projects
│   ├── css/
│   ├── images/
│   └── favicon.ico
├── project-alpha/
│   ├── assets/             ← project-specific (overrides shared)
│   ├── docmd.config.js
│   └── docs/
├── project-beta/
│   ├── docmd.config.js
│   └── v01/
└── docmd.config.js         ← root multi-project config
```

### Independent Versioning & i18n

Each project maintains its own version history and internationalisation setup. A documentation engine at v0.7 and a search tool at v0.1 can coexist under the same domain without version conflicts.

```javascript
// project-alpha/docmd.config.js
module.exports = defineConfig({
  title: 'Alpha Docs',
  versions: {
    current: '07',
    all: [
      { id: '07', dir: 'v07', label: 'v0.7' },
      { id: '06', dir: 'v06', label: 'v0.6' }
    ]
  }
});
```

### Multi-Project Dev Server

`docmd dev` in multi-project mode builds all projects and serves them from a single port. File changes in any project trigger a targeted rebuild with live reload.

```
────────────────────────────────────────
  MULTI-PROJECT DEV SERVER

  Local:    http://127.0.0.1:3000
  Network:  http://192.168.1.5:3000

  Project:  http://127.0.0.1:3000/ → project-alpha/
  Project:  http://127.0.0.1:3000/search → project-beta/
────────────────────────────────────────
```

### Graceful Configuration Validation
Configuration errors no longer leak Node.js stack traces. The validator now detects `projects` root configurations and provides clean, human-readable error messages for missing properties or typos, ensuring a premium CLI experience.

### Robust Developer Workflow
A new `pnpm prep` system has been implemented for monorepo developers. It features a "deep-wipe" mechanism that aggressively clears any global `docmd` binaries from the system before verification, ensuring that testing always happens against a fresh build.
- `pnpm prep`: Full reset and verification.
- `pnpm prep --link`: Full reset, verification, and global re-linking.

## 🔧 Architecture

The multi-project handler lives in `engine/projects.ts` — a standalone module that doesn't mix with other core files. It:

1. **Detects** `projects[]` in the root config before normal build starts
2. **Validates** prefixes (no duplicates, root `/` required)
3. **Iterates** over each project, `chdir`s into its directory
4. **Delegates** to the standard `buildSite()` with environment-injected `src`/`out`/`base`
5. **Merges** all outputs into a single `site/` directory

Zero changes to the existing build pipeline. The standard single-project workflow continues to work exactly as before.

## 🚀 Deployment

Multi-project output is a single static `site/` directory — compatible with GitHub Pages, Netlify, Vercel, Cloudflare Pages, and any static hosting. No nginx proxy configuration needed.

```yaml
# GitHub Actions (unchanged)
- run: npx docmd build
- uses: peaceiris/actions-gh-pages@v4
  with:
    publish_dir: ./site
```

## Migration Guide

**Existing single-project setups require no changes.** Multi-project mode only activates when a `projects` array is present in the root config.

To adopt multi-project:

1. Move your existing docs into a subdirectory (e.g., `my-docs/`)
2. Create a root `docmd.config.js` with `projects: [{ prefix: '/', src: 'my-docs' }]`
3. Remove `src` and `out` from the child config (the parent handles those)
4. Add more projects as needed

::: callout info
The root `docmd.config.js` should contain **only** the `projects` array. All other configuration (title, theme, plugins, etc.) belongs in each project's own `docmd.config.js`.
:::
