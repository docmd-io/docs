Das Release `docmd` 0.7.7 führt die **Multi-Projekt-Unterstützung** ein — die Möglichkeit, mehrere unabhängige Dokumentationsseiten aus einer einzigen docmd-Instanz zu erstellen und bereitzustellen. Ein Befehl, eine Domain, unabhängige Projekte.

## ✨ Highlights

### Multi-Projekt-Konfiguration

Eine einzelne `docmd.config.js` im Repository-Root kann nun mehrere Dokumentationsprojekte definieren. Jedes Projekt erhält sein eigenes Präfix, seine eigene Konfiguration, Versionierung und Navigation — nutzt aber dasselbe Theme, dieselben Assets und dieselbe Bereitstellungspipeline.

```javascript
// docmd.config.js (Root)
module.exports = defineConfig({
  projects: [
    { prefix: '/',       src: 'docmd-main' },
    { prefix: '/search', src: 'docmd-search' }
  ]
});
```

**Ausgabe:**
```
site/                     ← Root-Projekt (docmd-main)
site/search/              ← Unterpfad-Projekt (docmd-search)
```

Jeder Projektordner enthält seine eigene `docmd.config.js` mit vollständig unabhängiger Konfiguration — Titel, i18n, Versionen, Plugins, Navigation, Layout und alles andere.

Details finden Sie im Leitfaden [Multi-Projekt-Konfiguration](/configuration/multi-project/).

### Gemeinsame Assets

Assets im Repository-Root (`assets/`) werden automatisch an alle Projekte weitergegeben. Jedes Projekt kann auch ein eigenes `assets/`-Verzeichnis haben, das Vorrang vor gemeinsamen Assets hat.

### Unabhängige Versionierung & i18n

Jedes Projekt pflegt seine eigene Versionshistorie und Internationalisierung. Eine Dokumentations-Engine mit v0.7 und ein Such-Tool mit v0.1 können unter derselben Domain koexistieren, ohne Versionskonflikte.

### Multi-Projekt-Entwicklungsserver

`docmd dev` im Multi-Projekt-Modus erstellt alle Projekte und stellt sie über einen einzigen Port bereit. Dateiänderungen in einem Projekt lösen einen gezielten Neubau mit Live-Reload aus.

```
────────────────────────────────────────
  MULTI-PROJEKT-ENTWICKLUNGSSERVER

  Lokal:    http://127.0.0.1:3000
  Netzwerk: http://192.168.1.5:3000

  Projekt:  http://127.0.0.1:3000/ → project-alpha/
  Projekt:  http://127.0.0.1:3000/search → project-beta/
────────────────────────────────────────
```

## 🔧 Architektur

Der Multi-Projekt-Handler befindet sich in `engine/projects.ts` — ein eigenständiges Modul, das nicht mit anderen Core-Dateien vermischt wird. Er:

1. **Erkennt** `projects[]` in der Root-Konfiguration vor dem normalen Build
2. **Validiert** Präfixe (keine Duplikate, Root `/` erforderlich)
3. **Iteriert** über jedes Projekt, wechselt in dessen Verzeichnis
4. **Delegiert** an das Standard-`buildSite()` mit Umgebungsvariablen
5. **Vereint** alle Ausgaben in einem einzigen `site/`-Verzeichnis

Null Änderungen an der bestehenden Build-Pipeline. Der Standard-Einzelprojekt-Workflow funktioniert weiterhin genau wie zuvor.

## 🚀 Bereitstellung

Die Multi-Projekt-Ausgabe ist ein einzelnes statisches `site/`-Verzeichnis — kompatibel mit GitHub Pages, Netlify, Vercel, Cloudflare Pages und jedem statischen Hosting. Keine nginx-Proxy-Konfiguration erforderlich.

## Migrationsleitfaden

**Bestehende Einzelprojekt-Konfigurationen erfordern keine Änderungen.** Der Multi-Projekt-Modus wird nur aktiviert, wenn ein `projects`-Array in der Root-Konfiguration vorhanden ist.

::: callout info
Die Root-`docmd.config.js` sollte **nur** das `projects`-Array enthalten. Alle andere Konfiguration (Titel, Theme, Plugins usw.) gehört in die jeweilige `docmd.config.js` des Projekts.
:::
