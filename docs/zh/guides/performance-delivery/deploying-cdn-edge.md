## 问题
纽约和西雅图的中心单体服务器将把远在东京的用户物理延迟提高到了极限拉扯出几百毫秒级别的断崖落差。

## 为什么重要
长途跨越极度恶化“感知响应速度”。即使本框架 18KB 输出能力再小也难以逾越跨海通信光缆电信号极限。直接使弃文档概率剧增。

## 方法
寻找 Vercel 亦或 Netlify 这样的分布式多处境 Edge Server 在节点上做纯属静态路由的复制同步发布分发应用服务承接流量负载。

## 实施
纯原生导出的 HTML/JS/CSS 免去了配置障碍。利用 GH Actions (Github) 在编译环境写 `docmd build` 并指明将打包输出目标文件 `/site` 使用预制部署行为直接搬移到诸如 Cloudflare Pages 等无服务管理商环境即可。

## 权衡
分布式虽然解决了提速但遮掩了传统分析和运维能查询的统一集中日志平台（Log 面板）。对于区域单独掉线导致的离奇 bug，开发者较难诊断其全方位源头。
