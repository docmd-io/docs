## 问题
随便推送的主干合并会经常让文档站因格式引发全局的 500 报错或者断链等但是过度干预审核会让人没有纠错积极性。"

## 为什么重要
一个健康的平台通常依靠社区或公司全体随时的勘误如果没有快捷的在三十秒内搞定错别字修正路径这种勘误积极性就归零了。

## 方法
采用 Github 流的在线一键“Fork 然后发起 PR 推送拉取请求”的方式结合起来并辅之机器初步审视通过等组合拳来平衡体验。

## 实施
利用系统顶层的支持只需要 `editLink` 被激活 `docmd` 就能每页底部塞入类似 "在 Github 上编辑" 的按钮直达该页面的修改动作上。并能够协同类似插件去开展在指定行与部分位置去开战深度的批注式的代码审查（Code Review）流程体验效果及其实现手段细节等展开描述与呈现给你们参考实践。

## 权衡
因为所有的文档均要求严格执行 PR 这个制度那么有些譬如说服务临时熔断的公告说明等一些救火性质的公告更新就会因为这类审核而贻误战机无法极速发布。
