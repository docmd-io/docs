Das `docmd` 0.7.6 Release führt eine **einheitliche URL-Normalisierungs-Engine** ein, die 301-Weiterleitungen eliminiert, Trailing-Slash-URLs für SEO durchsetzt und eine konsistente Linkbehandlung über alle Oberflächen hinweg gewährleistet — Markdown-Inhalte, Button-Container, Tags und Navigationskonfiguration. Zwei neue universelle Link-Präfixe, `external:` und `raw:`, geben Autoren präzise Kontrolle über das Linkverhalten.

## 🔗 Einheitliche URL-Normalisierungs-Engine

### Das Problem

Vor 0.7.6 war die URL-Verarbeitung auf sechs separate Codestellen verteilt, mit subtilen Inkonsistenzen:

- **Fehlende Trailing Slashes** — Interne Links renderten als `/configuration/overview` statt `/configuration/overview/`, was 301-Weiterleitungen verursachte.
- **`index.md`-Artefakt** — Links wie `localisation/index.md` renderten als `localisation/index` statt `localisation/`.
- **Container-Blindstellen** — Button-Container und Tags reichten URLs ohne Normalisierung durch.
- **Doppelschrägstrich-Risiko** — Kein Schutz gegen `foo//bar/` bei versehentlichen Extra-Schrägstrichen.

### Die Lösung

Alle URL-Verarbeitung läuft nun über eine einzige `resolveHref()`-Funktion in `@docmd/parser`.

### URL-Übersetzungstabelle

| Eingabe | Ausgabe | Angewandte Regel |
|:---|:---|:---|
| `overview.md` | `overview/` | `.md` entfernt, Trailing `/` hinzugefügt |
| `overview` | `overview/` | Trailing `/` hinzugefügt |
| `overview/` | `overview/` | Bereits kanonisch |
| `localisation/index.md` | `localisation/` | `index.md` entfernt → Ordnerpfad |
| `../index.md` | `../` | Übergeordneter Verzeichnisindex |
| `overview.md#settings` | `overview/#settings` | Hash-Fragment beibehalten |
| `foo//bar.md` | `foo/bar/` | Doppelschrägstriche zusammengeführt |
| `https://example.com` | `https://example.com` | Extern — unverändert |
| `external:overview.md` | `overview/` *(+ neuer Tab)* | Präfix entfernt, Flags gesetzt |
| `raw:docs/readme.md` | `docs/readme.md` | Normalisierung umgangen |

## 🆕 Universelles `external:`-Präfix

Das neue `external:`-Präfix wird nun **überall** erkannt — Markdown, Buttons, Tags und alle Container.

```markdown
<!-- Link in neuem Tab öffnen -->
[Einstellungen öffnen](external:./configuration/overview.md)

<!-- HTTP-Links werden automatisch als extern erkannt -->
[GitHub](https://github.com/docmd-io/docmd)

<!-- Funktioniert auch in Buttons -->
::: button "API-Docs" external:./api/node-api.md icon:external-link
```

## 🆕 `raw:`-Präfix — Normalisierung umgehen

Da die Engine nun alle `.md`-Erweiterungen normalisiert, können Autoren mit dem `raw:`-Präfix auf Rohdateien verlinken:

```markdown
[Vorlage herunterladen](raw:templates/starter.md)

::: button "Quelle holen" raw:docs/readme.md icon:download
```

## 🚀 SEO-Verbesserungen

Alle internen Seiten rendern jetzt konsistent mit Trailing-Slash-URLs. Dies eliminiert die 301-Weiterleitungsketten.

## 📖 Dokumentation

Der [Verlinkung & Referenzierung](../content/syntax/linking.md) Leitfaden wurde umfassend überarbeitet. Die Dokumentation ist in allen drei Locales (Englisch, Chinesisch, Deutsch) verfügbar.

## 🔍 Umfassende Link-Fehlerbehebungen

Wir haben mehrere Randfall-Link-Verhaltensweisen behoben, die bei strengen Tests im gesamten Ökosystem festgestellt wurden:

- **Such-Client-Pfad**: Die Such-UI entfernt nun explizit führende Schrägstriche beim Erstellen der endgültigen `linkHref`, wodurch doppelte Schrägstriche (z.B. `//`) beim Navigieren zum Stammindex oder zu Überschriftenfragmenten verhindert werden.
- **Menüband-Normalisierung**: Die Menüband-Konfiguration (`config.menubar.left` und `config.menubar.right`) verwendet nun die Kern-URL-Normalisierungs-Engine (`resolveHref`). Dies stellt sicher, dass Menüband-Links genau dieselben Regeln für nachfolgende Schrägstriche und die Behandlung des `external:`-Präfixes befolgen wie Markdown-Links, wodurch 301-Weiterleitungen auf oberster Ebene eliminiert werden.
- **TOC- & Versionswechsler-Immunität gegen doppelte Schrägstriche**: Indem verhindert wird, dass der Such-Client die Adresszeile mit `//` verunreinigt, erben der Inhaltsverzeichnis-Generator (der auf relativen Hash-Links basiert) und der Versionswechsler (der den aktuellen Pfadnamen beibehält) keine doppelten Schrägstriche mehr und geben diese nicht weiter. Der Versionswechsler erhielt außerdem einen unabhängigen Regex-Schutz gegen doppelte Schrägstriche.
- **Site-Logo-Verlinkung**: Die Verknüpfungen für Site-Titel und Logo wurden aktualisiert, um auf das Stammverzeichnis (`./`) zu zielen, anstatt explizit auf `index.html` zu verlinken, wodurch eine Ausnahme in der sauberen URL-Struktur beseitigt wurde.

## Migrationsleitfaden

Für **Endbenutzer**: Aktualisieren Sie auf 0.7.6 mit `npm update @docmd/core` oder `pnpm update @docmd/core`. Keine Konfigurationsänderungen erforderlich.
