The `docmd` 0.7.6 release introduces a **unified URL normalisation engine** that eliminates 301 redirects, enforces trailing-slash URLs for SEO, and brings consistent link handling across every surface — Markdown content, button containers, tags, and navigation configuration. Two new universal link prefixes, `external:` and `raw:`, give users precise control over how links behave without learning container-specific syntax.

## 🔗 Unified URL Normalisation Engine

### The Problem

Prior to 0.7.6, URL handling was fragmented across six separate code locations. Each had its own inline logic with subtle inconsistencies:

- **Missing trailing slashes** — Internal links rendered as `/configuration/overview` instead of `/configuration/overview/`, causing web servers to issue 301 redirects to the canonical form.
- **`index.md` artefact** — Links like `localisation/index.md` rendered as `localisation/index` instead of the clean `localisation/`.
- **Container blindspots** — Button containers (`::: button`) and tags (`::: tag`) passed URLs through raw without any normalisation.
- **Double-slash risk** — No guard against `foo//bar/` if a user accidentally included an extra slash.

### The Fix

All URL handling is now routed through a single `resolveHref()` function in `@docmd/parser`. Every surface that generates or processes a link uses this function:

| Surface | Before 0.7.6 | After 0.7.6 |
|:---|:---|:---|
| Markdown links `[text](href)` | Inline logic, no external detection | `resolveHref()` with full feature set |
| Button containers `::: button` | Partial inline logic | `resolveHref()` with full feature set |
| Tag links `::: tag link:` | `startsWith('http')` only | `resolveHref()` with full feature set |
| Navigation config | Separate `normalizeNavPaths()` | Unified via `normalizeInternalHref()` |
| Auto-router (zero-config) | Inline regex | Unified via `normalizeInternalHref()` |
| Version filter | Manual reversal | Updated for trailing-slash paths |

### URL Translation Table

The normaliser handles every format a user might write and produces a consistent, SEO-optimised output:

| Input | Output | Rule Applied |
|:---|:---|:---|
| `overview.md` | `overview/` | Strip `.md`, add trailing `/` |
| `overview` | `overview/` | Add trailing `/` |
| `overview/` | `overview/` | Already canonical |
| `overview.html` | `overview/` | Strip `.html`, add trailing `/` |
| `api/commands.md` | `api/commands/` | Subdirectory normalised |
| `localisation/index.md` | `localisation/` | Strip `index.md` → folder path |
| `../index.md` | `../` | Parent directory index |
| `index.md` | *(empty — current dir root)* | Root index stripped |
| `./dir/index` | `./dir/` | Strip `/index` suffix |
| `overview.md#settings` | `overview/#settings` | Hash fragment preserved |
| `foo//bar.md` | `foo/bar/` | Double slashes collapsed |
| `https://example.com` | `https://example.com` | External — untouched |
| `#anchor` | `#anchor` | Hash-only — untouched |
| `/assets/file.pdf` | `/assets/file.pdf` | Asset path — untouched |
| `external:overview.md` | `overview/` *(+ new tab)* | Prefix stripped, flags set |
| `raw:docs/readme.md` | `docs/readme.md` | Normalisation bypassed |

## 🆕 Universal `external:` Prefix

A new `external:` prefix is now recognised **everywhere** a link can be written — Markdown, buttons, tags, and any container that renders links. Previously, forcing a new-tab link required different mechanisms in different contexts (or wasn't possible at all in standard Markdown).

### Before 0.7.6

| Surface | New-Tab Support |
|:---|:---|
| Markdown links | ❌ Not supported |
| Button containers | ✅ `external:` prefix |
| Tag links | ⚠️ Auto-detect `http` only |
| Navigation | ✅ `external: true` property |

### After 0.7.6

| Surface | New-Tab Support |
|:---|:---|
| Markdown links | ✅ `external:` prefix + auto-detect `http` |
| Button containers | ✅ `external:` prefix + auto-detect `http` |
| Tag links | ✅ `external:` prefix + auto-detect `http` |
| Navigation | ✅ `external: true` property (unchanged) |

### Usage

```markdown
<!-- Force any link to open in a new tab -->
[Open Settings](external:./configuration/overview.md)

<!-- HTTP links auto-detect as external -->
[GitHub](https://github.com/docmd-io/docmd)

<!-- Works in buttons too -->
::: button "API Docs" external:./api/node-api.md icon:external-link

<!-- And tags -->
::: tag "Source" link:external:./api/cli-commands.md icon:external-link
```

The `external:` prefix is stripped from the rendered URL — it is a build-time signal only.

## 🆕 `raw:` Prefix — Bypass Normalisation

Since the engine now normalises all `.md` extensions away, authors who genuinely need to link to a raw `.md` file (e.g., a downloadable source template) can use the `raw:` prefix:

```markdown
[Download Template](raw:templates/starter.md)

::: button "Get Source" raw:docs/readme.md icon:download
```

The `raw:` prefix bypasses all normalisation — the extension and path are preserved exactly as written. The prefix itself is stripped from the rendered URL.

## 🚀 SEO Improvements

### Trailing-Slash Enforcement

All internal pages now consistently render with trailing-slash URLs. This eliminates the 301 redirect chains that previously occurred when web servers (Nginx, Caddy, Cloudflare Pages) automatically appended the slash.

**Before 0.7.6:**
```
/configuration/overview     → 301 → /configuration/overview/
```

**After 0.7.6:**
```
/configuration/overview/    → 200 (direct hit, no redirect)
```

### Clean Canonical URLs

- All `<link rel="canonical">` tags use trailing-slash URLs.
- All `<link rel="alternate" hreflang>` tags use trailing-slash URLs.
- All sidebar and breadcrumb navigation links use trailing-slash URLs.
- Zero internal `.md` extensions in production HTML output.
- Zero `/index` artefacts in any rendered link.

## 🔍 Search — Root Page Fix

The search plugin generated page IDs from `outputPath` values. For the site root (`index.html`), the `.html` extension was stripped but the bare `index` was kept — producing a search result URL of `/index` instead of `/`. Clicking the homepage in search results would trigger a 301 redirect on most servers.

**Fix**: The root `index.html` is now correctly mapped to `/` in the search index. All search result links now match the canonical trailing-slash URLs used by the SEO, sitemap, and llms plugins.

## Migration Guide

**For end users**: Update to 0.7.6 with `npm update @docmd/core` or `pnpm update @docmd/core`. No configuration changes are required — the normalisation is fully backward compatible.

**Breaking changes**: None. All previously valid link formats continue to work. The only visible difference is that rendered URLs now consistently end with a trailing slash, which improves SEO and eliminates 301 redirects.

::: callout info
If you have custom server-side redirect rules that strip trailing slashes, you should remove them. The `docmd` engine now produces the canonical trailing-slash format natively.
:::
